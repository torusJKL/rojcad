## Context

rojcad uses `JANET_BOOTSTRAP=1` to build the Janet runtime from C source (vendored in `vendor/`) rather than from a pre-compiled bytecode image. This avoids a chicken-and-egg problem (the image must be generated by a working Janet compiler) and simplifies the embedded build.

The trade-off is that standard Janet macros (`defmacro`, `defn`, `->`, `each`, `for`, `loop`, `match`, `case`, etc.) are absent — they're normally defined in Janet source (`src/boot/boot.janet`) and compiled into the core image, which we don't have. Users at the REPL must use verbose alternatives like `(def name :macro (fn [...] ...))`.

The upstream Janet repo ships `src/boot/boot.janet` (164KB) alongside the C sources. This is the canonical definition of all standard Janet macros, written in Janet itself using only primitive special forms and registered C functions — both of which are fully available under `JANET_BOOTSTRAP`.

## Goals / Non-Goals

**Goals:**
- Provide the full standard Janet macro suite at the runtime REPL
- Zero reimplementation — use upstream source verbatim
- Fix the AGENTS.md documentation to reflect the new macro availability
- Keep rojcad's existing `boot.janet` unchanged and working

**Non-Goals:**
- Not rewriting rojcad's existing `boot.janet` to use macros (future work)
- Not fixing the `&form`/`&env` limitation (compiler-level, not fixable from Janet)
- Not adding new CAD-specific macros

## Decisions

1. **Embed upstream boot.janet as a separate file** rather than appending to rojcad's `boot.janet`.
   - *Why*: Upstream file is 164KB and independently maintained. A separate file makes diffing against upstream releases straightforward. Mixing with rojcad's 1839-line boot.janet would create a maintenance nightmare.
   - *Alternative considered*: Append to rojcad's boot.janet. Rejected because merging upstream updates would be error-prone.

2. **Load upstream file before rojcad's boot.janet**.
   - *Why*: Makes upstream macros available to rojcad's boot code if we choose to use them in the future. Also makes the initialization order clear: C bootstrap → upstream macros → rojcad CAD wrappers → REPL.
   - *Alternative considered*: Load after. This would mean rojcad's boot.janet can't use the macros, which is fine for now but less flexible.

3. **Fetch from upstream Janet's master branch** at JaneStreet's GitHub.
   - *Why*: The vendored Janet is v1.41.2. The upstream master branch at the time of implementation should match closely. A specific tag (e.g., v1.41.2) would be ideal if available.
   - *Risk*: If the upstream boot.janet changes incompatibly, rojcad would need to update. Mitigation: pin to the tag matching `vendor/janetconf.h`'s version.

4. **Use `include_str!` embedding** (same as rojcad's boot.janet).
   - *Why*: Already the established pattern. No new build dependencies. Zero runtime I/O.

5. **No changes to `boot.janet`**.
   - *Why*: It works as-is. Rewriting it to use macros is orthogonal work with higher risk. The `doc` name collision (upstream defines `doc` as a macro, rojcad as a function) resolves naturally — rojcad's definition wins because it loads second.

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| Upstream boot.janet changes incompatibly | Pin to the Janet version tag matching `vendor/janetconf.h`. Update both files in lockstep. |
| +164KB binary size increase | Acceptable — the boot.janet is text, compresses well, and the total binary is already large (OCCT). |
| Boot time increases | Parsing 164KB is fast (~ms). If it becomes an issue, pre-compile upstream.janet to bytecode in build.rs. |
| Name collision between upstream and rojcad definitions | Verified no harmful collisions. `doc` is the only overlap and rojcad's version wins (desired behavior). |
| Upstream boot.janet defines `defmacro` as a macro that needs `defn` — circular dependency | Not circular: `defn` is defined first using only special forms, then `defmacro` uses `defn`. The file ordering handles this. |

## Context

rojcad has two documentation outputs: a Janet API reference (`doc/janet-api.md` + `doc/janet-api.html`) generated by the headless binary at build time, and Rust crate docs (`target/doc/`) from `cargo doc`. Both exist only locally — users must build from source or download a tarball to view them. There is no hosted documentation site.

The existing Release workflow (`.github/workflows/release.yml`) builds a release binary, packages AppImage and tarball, and creates a GitHub Release — all triggered on `v*` tags. The `just tarball` step already runs `(dump-docs "doc")` during packaging, so Janet docs are generated for every release. Rust docs are not currently generated in CI.

No `gh-pages` branch exists yet. The repository has `contents: write` permission on the Release workflow.

## Goals / Non-Goals

**Goals:**
- Publish Janet API docs and Rust crate docs to GitHub Pages on each tagged release
- Versioned archive: each release creates `/<version>/` that persists forever
- Latest alias: `/latest/` overwritten each release for the "current" docs
- Zero extra compilation: reuse the release binary already built by the workflow
- Minimal workflow changes: add steps to existing `release.yml`, not a new file

**Non-Goals:**
- Custom domain, HTTPS configuration, or DNS setup (handled by GitHub Pages defaults)
- Search index, version selector UI, or doc-site chrome (raw file serving)
- Historical doc generation for past releases (only from this point forward)
- Doc generation from untagged commits or PR previews

## Decisions

### Decision: Extend existing release.yml rather than create a new workflow

**Chosen**: Add 4 steps to `release.yml`.

**Alternatives considered:**
- New `docs.yml` — would duplicate the entire OCCT + Rust build (~10+ min), doubling CI time and cache pressure. The release build already generates the binary — reusing it is the only efficient path.

### Decision: Use `peaceiris/actions-gh-pages@v4` for deployment

**Chosen**: This action is well-established, handles git commit/push to `gh-pages` branch, and its `keep_files: true` option naturally preserves old version directories on subsequent runs.

**Alternatives considered:**
- Manual git operations (checkout `gh-pages`, copy, commit, push) — more boilerplate, no advantage.
- Official `actions/upload-pages-artifact` + `actions/deploy-pages` — each deploy replaces the entire site; preserving old versions would require downloading the previous artifact, merging, and re-uploading. Significantly more complex.

### Decision: `keep_files: true` for archival

**Chosen**: `peaceiris/actions-gh-pages` with `keep_files: true` fetches the existing `gh-pages` branch, overlays the new `_site/` content, and leaves untouched files alone. First run (no branch yet) creates it fresh.

| Run | Tag | `_site/` contains | Result |
|---|---|---|---|
| 1st | v0.1.0 | `latest/`, `0.1.0/` | Branch created with both |
| 2nd | v0.2.0 | `latest/`, `0.2.0/` | `0.1.0/` kept, `0.2.0/` added, `latest/` updated |
| 3rd | v0.3.0 | `latest/`, `0.3.0/` | `0.1.0/`, `0.2.0/` kept, `0.3.0/` added, `latest/` updated |

### Decision: Janet API docs at root, Rust docs under `/rust/`

**Chosen**: Maintaining backward-compatible URLs (`/latest/janet-api.html` stays the same) and treating Rust docs as supplementary.

| Prefix | Contents |
|---|---|
| `/latest/janet-api.html` | Janet API (HTML) |
| `/latest/janet-api.md` | Janet API (Markdown) |
| `/latest/rust/` | Full `target/doc/*` (cargo doc output) |
| `/<version>/janet-api.html` | Same structure, versioned |
| `/<version>/rust/` | |

### Decision: Use `GITHUB_TOKEN` for authentication

**Chosen**: The existing `permissions: contents: write` in `release.yml` is sufficient for `peaceiris/actions-gh-pages` to push to the `gh-pages` branch within the same repository. No separate deploy key or personal access token needed.

## Risks / Trade-offs

| Risk | Mitigation |
|---|---|
| **First run creates `gh-pages` branch** — may need GitHub Pages to be enabled in repo settings | Add a note to release notes / setup instructions. The action creates the branch, but GitHub Pages must be configured to serve from it. |
| **`keep_files: true` and file deletion** — if a file needs to be removed from an old version, it won't happen automatically | Acceptable — old versions are archival. Manual cleanup via a separate commit if needed. |
| **Rust doc generation adds ~30-60s to CI** — `cargo doc --no-deps` recompiles the crate with doc metadata | Incremental compilation keeps this fast. Acceptable for a tagged release workflow that already takes 10+ min (OCCT). |
| **`GITHUB_TOKEN` rate limits** — Pages pushes from GHA use the standard token | Low risk — at most one push per release tag. |
